### How does itÂ work?

In essence we are creating an object called **list**. This is the base of our Linked â€˜**List**â€.

The **list** object has a key called **head**. The special thing about the **head** is that it does not hold a value, but is just a **pointer** to what actually holds the value. The continuing nested objects will contain values and pointers until the last element points to **null.** If you need a reminder on how referencing works, check out this helpful [blog](https://daveceddia.com/javascript-references/).

So we have a bunch of nested objects (and the point here is to realize that that is just what they are, objects) and to clarify what their purpose we name them after what they do.

#### Tie primitive with structure

Each nested object is referred to as a Node, and it is easy to think of these Nodes as **a list with the length of 1**, and the process of setting a _pointer_ is how we _link_ them together, otherwise they would just be different objects floating around in memory somewhere, _like a ship on open water._

**Using ES6** class function we can now easily create linked lists.

We can create functions that create **nodes** for us.

We can use the full power of classes to set **methods** that will give us special powers over the _linked list_.

It is with these **methods** that we will _manipulate_ the linked list. Adding a **node** or removing a node, printing the list, or getting the length of the list.

### Practical Applications

The point behind this blog is not about the HOW, but the **WHAT,** and **WHY** but when you understand the W&W you can start to see the HOW and where linked lists are being used.

Because of its abilities ofÂ :

- Fast insertion / removal
- Fast access to first element (and end if implemented bidirectional)
- Efficient memory usage
- Dynamic storage

**Practical Applications** of a _linked list_ include building the base for other more controlled data structures, specifically the **Queue,** and the **Stack.** Each follow a design paradigm for controlling how the data is accessed, the Queue being _first in, first out (_**_FIFO_**_)_ and the Stack being _first in, last out (_**_FILO_**_)._ I am planning on covering these in upcoming articles.
